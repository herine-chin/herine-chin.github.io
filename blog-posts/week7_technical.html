SQL vs. NoSQL

August 10, 2014

SQL

When I mention SQL in this post, it's not specifically the language but the databases it's used to access, also called relational databases. This was introduced in the 1970s and uses a mathmatical approach to structuring data. It is a set of tables containing data that fits into predefined fields. Each table can have one or many data categories in the form of columns with each row representing a unique instance of data that correlates with the categories in the columns.

PROs:
Relational databases can be queried with a standard query language like SQL. It is more reliable and has more support due to it's age and open source nature.

CONs:
The JOIN operation can slow systems down when there are many users (think millions) are doing lookups against tables with lots (think millions again) of data. In order to scale a relational database it has to be distributed across multiple servers and handling tables across different servers is chaos. If your data doesn't fit into tables easily then it requires you to design your database into a more complex structure to work.

NoSQL

NoSQL is a newer type of non-relational database, also called structured storage database. It doesn't require a schema, avoids join operations and typically scales horizontally. There are three types of popular architecture; key-value, column-oriented and documentbased stores.

PROS:
Data can be inserted without defining database schemas which means the format can be changed without application disruption, adding to its flexibility. It can also spread data across multiple servers without application assistance. It can process data faster because data models are typically simpler.

CONs:
Due to its age, NoSQL has less support and a smaller community to rely on for help. It also lacks a standard querying language, like SQL, because each database can be implemented differently.

What's next?

I look forward to working with NoSQL databases in the future. It doesn't mean I'm a proponent for it. I have some experience with relational database management and it would be great to learn from both sides. From my research it seems that NoSQL provides a type of flexibility needed in the modern age of projects but it's lacking the important standard language. I feel that NoSQL databases would take over relational databases once a querying langage has been achieved.
